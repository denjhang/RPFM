# RPFM 架构设计文档

> 记录时间：2026-05-11
> 硬件：RP2350A 核心板 + SPFM 声卡模块 (YM2413)
> 目标：多芯片 FM 音源播放器下位机

## 1. 系统定位

RPFM 是 **下位机**，专门负责音频芯片驱动。不处理文件系统、不管理存储、不跑复杂 UI。

```
PC/ESP32 (上位机) ──→ RPFM (下位机) ──→ YM2413/YM2151
```

上位机负责：文件管理、UI 交互、WiFi/Bluetooth 通信
下位机负责：SPFM 总线驱动、PIO 时序、实时音频流处理

## 2. 通信协议

### 2.1 方案对比

| 方式 | 速度 | 引脚 | 说明 |
|------|------|------|------|
| USB CDC | 12Mbps | D+/D- | 当前在用，适合开发调试 |
| UART 1.5M | 1.5M baud | TX/RX = 2 | 兼容 SPFM 协议，直接复用上位机 |
| SPI | 10M+ | SCK/MOSI/MISO/CS = 4 | 适合大数据量流式传输 |

### 2.2 推荐方案

**UART 1.5M**：与 SPFM 原有协议兼容，Denjhang Music Player 上位机无需改动。VGM 数据包通过 UART 发送到 RP2350，RP2350 解码后写入音频芯片。

后续可升级到 **SPI** 以实现更高带宽（如多通道 PCM 混音实时流）。

## 3. 存储架构

### 3.1 板载 QSPI Flash（16MB）

- 非 PSRAM，不能当运行内存用
- BOOTSEL 模式下显示 128MB 是 USB MSC 协议的虚拟大小，实际物理容量 16MB
- 适合存固件代码（~100KB）+ 预置测试曲（~1MB）
- 运行时不能擦写 XIP 区域，数据分区需要代码拷贝到 RAM 再写
- 10 万次擦写寿命，不适合频繁写入

### 3.2 外挂 PSRAM（未来扩展）

| 型号 | 容量 | 接口 |
|------|------|------|
| ESP-PSRAM64H | 64MB | SPI/QPI |

PSRAM 是真正的运行内存，用于：
- YM2612 PCM 数据加载缓冲
- VGM 流式播放环形缓冲
- ILI9341 帧缓冲（320x240x2 = 150KB）
- 多通道混音中间数据

### 3.3 结论

当前阶段不需要 PSRAM，RP2350A 的 **520KB SRAM** 足够做 VGM 环形缓冲（128KB）+ 解码缓冲。PSRAM 在加 YM2612 PCM 或混音之后再考虑。

```
SRAM 分配（当前方案）：
  ┌─────────────────────────┐
  │ 固件代码 + 变量  ~100KB │
  │ VGM 环形缓冲    ~128KB  │
  │ 解码缓冲         ~64KB  │
  │ 剩余             ~228KB │
  └─────────────────────────┘
```

不设 SD 卡，数据全部通过 USB/UART 实时流式传输。

## 4. PIO 驱动设计

### 4.1 核心要求：连续 GPIO

PIO 指令 `out pins, N` 要求 **GPIO 编号连续**，不是物理引脚连续。

当前 YM2413 需要 11 个连续 GPIO（D0-D7 + WR# + A0 + CS0#）：

```c
out pins, 11  // 需要 GPIO[N] 到 GPIO[N+10]
```

### 4.2 各板连续 GPIO 情况

| 板卡 | 连续 11 个空闲 GPIO | 说明 |
|------|---------------------|------|
| RP2350A 核心板 | ✅ GPIO0-10 | 目前面包板方案 |
| RP2350 Touch LCD | ❌ 最长连续段 GPIO0-4（5 个） | GPIO 分散 |
| ESP32S3 | ❌ | 非 PIO 架构 |

### 4.3 当前接线（RP2350A 核心板）

```
GPIO0-7   → D0-D7  数据总线  (PIO out, 11-bit 并行)
GPIO8     → WR#    写信号    (PIO out)
GPIO9     → A0     地址选择  (PIO out)
GPIO10    → CS0#   片选      (PIO out)
GPIO11    → IC#    复位      (CPU)
GPIO12-14 → A1-A3  保留      (CPU)
GPIO25    → LED               (CPU)
```

### 4.4 SPI 总线（未来共用）

ILI9341 屏幕、外挂 PSRAM、调试 UART 等外设可以共用一个 SPI 总线：

```
SPI0:
  SCK      → GPIO18（或任意空闲引脚）
  MOSI/TX  → GPIO19
  MISO/RX  → GPIO16
  CS_LCD   → GPIO15
  CS_PSRAM → GPIO17
  DC       → GPIO20
  RST      → GPIO21
```

## 5. 多芯片支持

### 5.1 SPFM 总线架构

SPFM 总线的关键在于**复用数据总线，独立片选**：

```
数据总线 (D0-D7) + WR#：所有芯片共用
A0:          YM2413 地址选择
CS0:         YM2413 片选
CS1:         YM2151 片选（未来）
A1:          YM2608 地址选择（未来）
```

PIO 程序只需写一次 WR# 脉冲逻辑，不同芯片的区别仅在 C 侧控制的 CS/A0 引脚。

### 5.2 支持计划

| 芯片 | 状态 | 说明 |
|------|------|------|
| YM2413 (OPLL) | ✅ 已驱动 | 9 通道旋律 + 5 节奏 |
| YM2151 (OPM) | 🔜 计划中 | 8 通道 FM，需注意初始化时序 |
| YM2612 (OPN2) | 🔜 计划中 | 6 通道 FM + DAC/PCM，需要 PSRAM |
| YM2203 (OPN) | 🔜 计划中 | 3 FM + 3 SSG，兼容 YM2151 类似 |
| YM2608 (OPNA) | 🔜 计划中 | 6 FM + 3 SSG + ADPCM |

## 6. 节奏通道（YM2413）

参考 libvgm 模拟器核心代码 `ym2413.c` 确认的 bit 映射：

```
寄存器 0x0E:
  bit 5 (0x20): 节奏模式启用
  bit 4 (0x10): Bass Drum  (BD)
  bit 3 (0x08): Snare Drum (SD)
  bit 2 (0x04): Tom Tom    (TOM)
  bit 1 (0x02): Top Cymbal (CYM)
  bit 0 (0x01): Hi-Hat     (HH)
```

触发节奏时只需写 0x0E 对应 bit 位即可，YM2413 内部自动从节奏乐器 ROM 加载参数。节奏音量由寄存器 0x36-0x38 控制。

## 7. 调试输出

### 7.1 USB CDC + BOOTSEL 命令

- 所有 `printf()` 输出走 USB CDC（TinyUSB 0.20.0，已替换 SDK 内置版）
- `getchar_timeout_us(0)` 轮询 CDC 输入
- 收到 `BOOTSEL\r\n` 调用 `rom_reset_usb_boot(0, 0)` 进入 BOOTSEL 模式
- 自动烧录脚本 `flash.sh`：发 BOOTSEL → 等盘符 → 复制 UF2

### 7.2 屏幕调试信息（未来）

通过 ILI9341 显示：
- 当前播放通道和音符
- 乐器编号
- 音量电平表
- 缓冲区状态（环形缓冲占用率）
- 最后调试信息

## 8. 上位机通信架构

### 8.1 通信协议

```
上位机 → 下位机：
  [CMD 1字节] [数据 N字节]

  CMD 列表：
    0x01: 写入寄存器 (reg, data, chip)
    0x02: 开始流式传输
    0x03: 停止/复位
    0x04: 设置音乐速率（如果是跳帧）
    0x10: 请求状态（当前缓冲区占用等）

下位机 → 上位机：
  [状态 1字节] [数据 N字节]

  状态：
    0x00: 正常 / 缓冲区空闲
    0x01: 缓冲区满 / 忙
    0x10: 调试信息打印
```

### 8.2 时序要求

VGM 播放的实际时序由下位机控制。上位机发送数据包后，下位机按 YM2413 的时钟（~3.58MHz）计算等待时间。

```
上位机发送延迟 → 下位机接收 → 写入芯片 → sleep_us(等待) → 下个数据
```

下位机的 `sleep_us()` 必须精确（PIO + busy_wait_us_32），上位机只需保证数据包在缓冲区溢出前送达。

## 9. 多板协作方案

### 9.1 场景：PC + RP2350A（调试阶段，当前）

```
PC ──USB CDC──→ RP2350A ──PIO──→ YM2413
```

PC 运行 Denjhang Music Player，通过 USB CDC 发送 VGM 数据。
下位机只负责解码和时序驱动。

### 9.2 场景：ESP32S3 + RP2350A（无线播放）

```
iPad/手机 ──WiFi──→ ESP32S3 ──UART 1.5M──→ RP2350A ──PIO──→ YM2413
                         │
                         └──触摸屏可视化
```

ESP32S3 做文件服务器 + WiFi MIDI 接收 + 触摸 UI

### 9.3 场景：ESP32P4 + RP2350A（旗舰独立播放器）

```
ESP32P4 ──SPI──→ RP2350A ──PIO──→ YM2151/YM2413
   │                      │
   ├── HDMI/USB 大屏      └── PSRAM (VGM 缓冲)
   ├── SD 卡 (文件库)
   └── PSRAM (混音)
```

ESP32P4 做主控，RP2350A 做纯音频驱动。

### 9.4 场景：RP2350 Touch LCD 做显示终端

```
RP2350A (音频) ──UART──→ RP2350 Touch LCD (显示)
```

RP2350 Touch LCD 的 GPIO 分散（无 9 个连续空闲），不适合 PIO 音频驱动。
专注做可视化接收：电平表、音符信息、音量柱状图。

## 10. GPIO 需求总表（完整方案）

| 外设 | GPIO 数 | 连续性要求 |
|------|---------|-----------|
| PIO 总线 D0-D7 + WR# + A0 + CS0# | 11 | 必须连续 |
| 控制信号 IC#, CS1 | 2 | 不要求 |
| SPI 总线 (SCK, TX, RX) | 3 | 不要求 |
| 外设片选 (LCD, PSRAM 等) | 每个设备 1 | 不要求 |
| UART TX/RX | 2 | 不要求 |
| LED | 1 | 不要求 |
| **合计** | **~20** | |

RP2350A（48 GPIO）完全充裕。
