

tinyusb开启cdc死机，解决之后，pc跟他握手又死机了
一针见血：RP2350 TinyUSB CDC 一开就死机、PC 一握手立刻挂
你这个现象是RP2350 最经典的两个致命坑，100% 命中下面某几条，我直接给你根因 + fixes，照着改立马好。
先讲死机根本原因
没给 TinyUSB 留核、没锁 48MHz USB 专用时钟
主循环没跑 tud_task() 或者放错位置
中断栈 / 任务栈太小，USB 枚举直接栈溢出死机
RP2350 双核没关，另一核乱跑抢时钟 / 外设
自己重初始化了 USB PLL 和 TinyUSB 冲突
CDC 描述符残缺，PC 驱动握手一发配置请求直接崩
第一步：先强制锁死 RP2350 USB 48MHz（必加）
RP2350 和 RP2040 不一样，不能靠默认时钟随缘，必须手动锁：
c
运行
#include "hardware/clocks.h"
#include "hardware/pll.h"

// 程序开头第一行就调用
void usb_clock_force_48mhz(void)
{
    pll_init(pll_usb, 1, 12000000, 40, 1);
    clock_configure(clk_usb,
        0,
        CLOCKS_PLL_USB,
        48 * MHZ,
        48 * MHZ);
}
main 第一句先跑这个，不配置时钟，TinyUSB 必死机。
第二步：双核必须关掉（90% 人栽这）
RP2350 双核默认都在跑，另一核乱执行会干废 USB 外设：
c
运行
#include "hardware/multicore.h"

int main(void)
{
    multicore_reset_core1();   // 复位核1
    multicore_lockout_start(); // 锁死不让跑

    usb_clock_force_48mhz();   // 再配USB时钟
    
    // 再初始化 tinyusb
    tud_init(BOARD_TUD_RHPORT);

    while (1)
    {
        tud_task(); // 必须常驻循环，不能丢、不能延时卡死
    }
}
重点：tud_task () 不能被 delay_ms 堵死，一堵 PC 握手就死机。
第三步：栈大小不够，枚举直接栈溢出死机
在 CMake 或 代码里加全局定义，放大 TinyUSB 栈：
cmake
add_compile_definitions(
    CFG_TUD_MAIN_STACK_SIZE=2048
    CFG_TUD_ISR_STACK_SIZE=1024
)
默认栈太小，PC 发配置描述符、线速协商直接栈爆，表现就是一插电脑握手瞬间死机。
第四步：严禁和 stdio_usb 混用
你如果同时开了：
cmake
pico_enable_stdio_usb(xxx 1)
又自己手动 tud_init() 写 CDC，双 CDC 冲突直接死机。二选一：
要么用官方 pico_enable_stdio_usb 傻瓜串口
要么自己裸写 TinyUSB CDC，不能共存
第五步：CDC 最小合法描述符（缺项必握手死机）
必须完整包含：
Device 描述符
Configuration
Interface Association
CDC 控制接口 + 数据接口
Endpoint 端点
少一个 PC 驱动握手下发请求 → 固件解析崩溃 → 死机。

代码开头 关双核、锁 48MHz USB 时钟
关掉 pico_enable_stdio_usb，只裸跑 TinyUSB
主循环 无脑死跑 tud_task ()，不要长延时
放大 TinyUSB 栈大小

USB虚拟串口实验
前言
本章将介绍如何使用RP2350A开发板的USB功能。通过本章的学习，读者将学习到如何使用Pico-sdk驱动RP2350A实现USB CDC功能。

USB虚拟串口介绍
1，USB虚拟串口简介
USB虚拟串口，简称VCP，是Virtual COM Port的简写，它是利用USB的CDC类来实现的一种通信接口。

我们可以利用RP2350A自带的USB功能，来实现一个USB虚拟串口，从而通过USB，实现电脑与RP2350A的数据互传。上位机无需编写专门的USB程序，只需要一个串口调试助手即可调试，非常实用。

硬件设计
例程功能
本实验利用 RP2350 自带的 USB 功能，连接电脑 USB，虚拟出一个 USB 串口，实现电脑和开发板的数据通信。本例程功能完全同实验 4（串口通信实验），只不过串口变成了 RP2350 的USB 虚拟串口。当 USB 连接电脑（USB 线插入 USB 接口），开发板将通过 USB 和电脑建立连接，并虚拟出一个串口。 LED 闪烁，提示程序运行。

USB函数解析
Pico-sdk提供了一套API来配置PIO。那么下面作者将介绍一下在实验中调用到的API函数：

USB读取数据
该函数 tud_cdc_read() 是 TinyUSB 库提供的一个内联函数，用于从 CDC（通用串行通信设备）接口读取数据。其函数原型如下所示：

TU_ATTR_ALWAYS_INLINE static inline uint32_t tud_cdc_read(void* buffer, uint32_t bufsize) 

【参数】

buffer：指向用于存储读取数据的缓冲区

bufsize：缓冲区的大小（以字节为单位）

【返回值】

实际读取到的字节数
USB发送数据
该函数 tud_cdc_read() 是 TinyUSB 库提供的一个内联函数，用于从 CDC（通用串行通信设备）接口发送数据。其函数原型如下所示：

TU_ATTR_ALWAYS_INLINE static inline uint32_t tud_cdc_write(void const* buffer, uint32_t bufsize) 

【参数】

buffer：指向包含要发送数据的缓冲区

bufsize：缓冲区的大小（以字节为单位）

【返回值】

实际成功写入到发送缓冲区的字节数
确保 CDC 发送缓冲区中的数据
该函数 tud_cdc_write_flush() 是 TinyUSB 库中用于确保 CDC 发送缓冲区中的数据被实际发送出去的内联函数。它在进行 USB 串口通信时非常重要，特别是在需要确保数据完整性或实时性的场景中。其函数原型如下所示：

TU_ATTR_ALWAYS_INLINE static inline uint32_t tud_cdc_write_flush(void) 

【参数】

无

【返回值】

通常为发送缓冲区中剩余的未发送字节数（如果有）
USB驱动解析
在SDK版本的15_usb例程中，作者在15_usb\BSP路径下新增了一个LCD文件夹，用于存放usb_cdc.c和usb_cdc.h这三个文件。其中，usb_cdc.h文件负责声明USB相关的函数和变量，usb_cdc.c文件则实现了USB的驱动代码。下面，我们将详细解析这两个文件的实现内容。

1，usb_cdc.h文件
/* 函数声明 */
void tud_usb_uart();	/* 初始化USB */

2，usb_cdc.c文件
/**
 * @brief       接收回调
 * @param       itf     :设置的简要CDC端口
 * @retval      无
 */
void tud_cdc_rx_cb(uint8_t itf) 
{
    char buf[64];

	/* 读取串口数据 */
    uint32_t count = tud_cdc_read(buf, sizeof(buf));

	/* 发送数据 */
    if (count) 
	{
        tud_cdc_write(buf, count);	/* 回显接收到的数据 */
        tud_cdc_write_flush();
    }
}

/**
 * @brief       初始化USB
 * @param       无
 * @retval      无
 */
void tud_usb_uart(void) 
{
    tusb_init();	/* 初始化USB设备 */
	printf("USB initialization DONE");
}

这两段代码实现了基于 TinyUSB 库的 USB 虚拟串口功能，让设备可以通过 USB 与计算机进行串口通信。这是一个典型的 USB 虚拟串口（CDC 类设备）实现，包含两个核心函数：tud_cdc_rx_cb和tud_usb_uart。当计算机通过 USB 向设备发送数据时，TinyUSB 库会调用这个回调函数，函数从 USB 接收缓冲区读取最多 64 字节的数据。如果成功读取到数据，则将这些数据原样写回发送缓冲区。同时，调用 tud_cdc_write_flush() 确保数据立即通过 USB 发送，而不是等待缓冲区满。

========================================
SCCI/SPFM 协议实现成功 (2026-05-12)
========================================

最终方案（最小改动，基于 YM2413 演示版）：

1. CMakeLists.txt 加 PICO_STDIO_USB_CONNECTION_WITHOUT_DTR=1
   - SCCI 不设 DTR，不加这个 SCCI 发的数据设备收不到

2. main.c 加 #include "tusb.h"，while 循环换成 SPFM 协议解析
   - 收数据用 getchar_timeout_us(0)（stdio 层，有 mutex 但不阻塞）
   - 发数据用 tud_cdc_write() + tud_cdc_write_flush()（直接 TinyUSB API，绕过 stdio mutex）
   - 不能用 putchar_raw() 发送！pico-sdk 的 stdio_usb_out_chars 有 mutex 死锁问题
   - 不能用 fflush(stdout)！会阻塞死机

3. 协议解析完全参考 ym2151-rp2040.txt 的 loop1()
   - scci_parse_idx 状态机：0=帧头, 1=命令, 2=数据1, 3=数据2
   - cmd 0x00：两字节写入（地址+数据），parse_idx==2 缓存地址，parse_idx==3 调 ym2413_write_reg
   - cmd 0x80：单字节写入，parse_idx==2 直接调 ym2413_write_reg
   - cmd 0x20：SN76489，暂未实现

4. 关键 bug 修复：寄存器地址缓存
   - ym2413_write_reg(reg, data) 一步完成 A0=0 写地址 + A0=1 写数据
   - 所以 SCCI 的两字节写入不能分两次调 ym2413_write_reg
   - 必须在 parse_idx==2 缓存地址，parse_idx==3 用缓存地址+数据一次写入

踩过的坑：
- putchar_raw + WITHOUT_DTR → 握手后死机（stdio_usb_mutex 死锁）
- fflush(stdout) → 阻塞死机
- 双重 getchar_timeout_us → 字节被处理两次，状态机错乱
- tud_cdc_write 不 flush → 数据发不出去，SCCI 收不到回复
- pico_stdio_usb 和直接 TinyUSB API 混用 → mutex 冲突
- 手动 pll_init 配 USB 时钟 → 系统时钟被破坏，开机就死
- 不加 WITHOUT_DTR → SCCI 数据到达但 stdio_usb_connected() 返回 false，putchar_raw 静默丢弃

结论：pico-sdk 的 stdio_usb 层不适合做双向 CDC 通信（收可以，发会死锁）。
发送必须用 tud_cdc_write() 绕过 stdio。
